# Project Goals: PQS RTN Hybrid Storage System

## 1. Project Vision (วิสัยทัศน์โครงการ)
พัฒนา Desktop Application เพื่อเป็นเครื่องมือสำหรับ "ผู้สร้าง (Creator)" ในการจัดทำชุดเอกสารมาตรฐานคุณวุฒิ (PQS) ของกองทัพเรือ โดยเน้นความง่ายในการใช้งาน (User-Friendly), ความยืดหยุ่นของข้อมูล (Data Portability), และรองรับกระบวนการสอบทั้งรูปแบบ Offline และ Online

## 2. Core Objectives (เป้าหมายหลัก)

### 2.1 Smart Content Editor (ระบบสร้างเอกสาร)
*   **Intuitive UX/UI:** ออกแบบหน้าจอให้ผู้สร้าง (Creator) สามารถป้อนข้อมูลได้ง่ายที่สุด โดยรองรับโครงสร้างข้อมูลแบบลำดับชั้น (Hierarchical Data):
    *   คำถามหลัก (Main Questions)
    *   คำถามรอง (Sub-questions)
    *   คำถามย่อย (Level 2 Sub-questions)
    *   คำตอบและเกณฑ์การให้คะแนน
*   **Visual Preview:** แสดงผลตัวอย่างเอกสารได้ทันทีขณะแก้ไข (WYSIWYG) โดยอ้างอิงรูปแบบจากไฟล์ตัวอย่าง (เช่น `202Lcp.tsx`)

### 2.2 Data Management & Portability (การจัดการฐานข้อมูล)
*   **Hybrid Storage:** เก็บข้อมูลในรูปแบบ Database ที่มีโครงสร้างชัดเจน เพื่อให้สามารถนำกลับมาแสดงผลและแก้ไขได้ถูกต้อง 100%
*   **Portable Database:** ระบบต้องรองรับการ Import/Export ฐานข้อมูลของ "แต่ละเอกสาร" (Document-specific DB) เพื่อให้สามารถส่งต่อไฟล์ PQS ที่สร้างเสร็จแล้วไปใช้งานบน Desktop App เครื่องอื่นได้โดยไม่มีความผิดเพี้ยน

### 2.3 Documentation Output (การออกรายงาน)
รองรับการพิมพ์หรือบันทึกเป็น PDF สำหรับการสอบแบบ Offline (Qualifying) โดยแยกรูปแบบอัตโนมัติ:
*   **Trainee Edition (เล่มคำถาม):** สำหรับผู้รับการทดสอบ แสดงเฉพาะโจทย์และช่องว่างสำหรับเขียนคำตอบ
*   **Qualifier Edition (เล่มเฉลย/คู่มือ):** สำหรับกรรมการผู้ทดสอบ แสดงโจทย์พร้อมแนวทางคำตอบและเกณฑ์การให้คะแนน

### 2.4 Hybrid Assessment (ระบบการทดสอบ)
*   **Offline Mode:** รองรับกระบวนการสอบแบบกระดาษ (ตามข้อ 2.3)
*   **Online Mode:** รองรับการทำข้อสอบผ่านระบบคอมพิวเตอร์สำหรับ "หัวข้อที่กำหนด" (Pre-defined Online Questions) โดยสามารถบันทึกและประมวลผลคะแนนได้

### 2.5 Centralized Reference System (ระบบเอกสารอ้างอิงส่วนกลาง)
*   **Concept:** จัดทำฐานข้อมูลเอกสารอ้างอิงกลาง (Master Reference Database) ที่ทุกชุดเอกสาร PQS สามารถดึงรายการไปใช้งานร่วมกันได้ (Reusable / One-to-Many) ลดความซ้ำซ้อนของการป้อนข้อมูล
*   **Security Classification (การจัดการชั้นความลับ):**
    | ชั้นความลับ (ไทย) | English | ความหมาย | สิทธิ์การเข้าถึง |
    | :--- | :--- | :--- | :--- |
    | **ลับที่สุด** | Top Secret | กระทบความมั่นคงร้ายแรงที่สุด | ⛔ ห้าม Download |
    | **ลับมาก** | Secret | กระทบความมั่นคงร้ายแรง | ⛔ ห้าม Download |
    | **ลับ** | Confidential | กระทบความมั่นคง | ⛔ ห้าม Download |
    | **ปกปิด** | Restricted | จำกัดเพื่อป้องกันผลเสีย | ✅ Download ได้ (เฉพาะผู้มีสิทธิ์ Trainee+) |
    | **ไม่ลับ** | Unclassified | เปิดเผยได้ | ✅ Download ได้ทั่วไป |
*   **Creator UX/UI:**
    *   **Search & Select:** มีระบบค้นหาเอกสารอ้างอิงจากฐานข้อมูลกลางเพื่อนำมาใส่ใน PQS
    *   **Quick Add:** กรณีไม่พบข้อมูล สามารถกด "เพิ่มเอกสารอ้างอิงใหม่" (Popup Form) ได้ทันทีระหว่างทำรายการ
    *   **Auto-Ordering & Numbering:** ระบบจะจัดเรียงรายการเอกสารอ้างอิงที่เลือกมา **ตามความยาวของชื่อเอกสาร (สั้นไปยาว) โดยอัตโนมัติ** เพื่อความสวยงาม และรันลำดับหัวข้อด้วย **ตัวอักษรไทย (ก., ข., ...)** ให้ทันที

### 2.6 Advanced Question Logic (ระบบจัดการคำถามซับซ้อน) [NEW]
*   **Concept:** รองรับโครงสร้างคำถามที่มี "ชุดคำถามย่อย (Pooled Sub-questions)" ที่กำหนดไว้ล่วงหน้าสำหรับแต่ละประเภทหัวข้อ (Generic Sections)
*   **Logic (SelectedSubQuestions):**
    *   **Predefined Pool:** แต่ละหัวข้อหลัก (เช่น "ส่วนประกอบและชิ้นส่วน...", "ค่าทำงานปกติ...") จะมีชุดคำถามย่อยเตรียมไว้ให้ (เช่น "มีหน้าที่อะไร", "ติดตั้งที่ไหน", "ใช้พลังงานอะไร")
    *   **Multi-Selection:** ผู้สร้างสามารถเลือกคำถามย่อยจาก Pool นี้ได้มากกว่า 1 ข้อ เพื่อใช้กับรายการนั้นๆ
    *   **Section Consistency:** ในหัวข้อ Section หนึ่งๆ (เช่น 202.2.1 ถึง 202.2.x) จะต้องแสดงผลเฉพาะคำถามยอดฮิตที่ถูกเลือกมาใช้งานจริงเท่านั้น (ไม่แสดงตัวที่ไม่ได้เลือก)
*   **Answer Binding (การเชื่อมโยงคำตอบ):**
    *   คำตอบแต่ละข้อต้องผูกติดกับคำถามย่อยที่เลือก (1-to-1 Mapping)
    *   **UI Requirement:** ในกล่องคำตอบ ต้องมีการทวนคำถาม หรือแสดง Label (ก., ข., ค.) ให้ตรงกับคำถามย่อยที่เลือกเสมอ
*   **Rendering (การแสดงผล):**
    *   ต้องแสดง Checkbox หรือสัญลักษณ์ที่ระบุว่าคำถามย่อยข้อใดถูกเลือกใช้งานบ้างในหัวข้อนั้นๆ เพื่อให้เอกสารมีความชัดเจนตามแบบ Example

## 3. User Roles (ผู้ใช้งานระบบ)
1.  **Creator:** ผู้สร้างและแก้ไขเนื้อหาชุด PQS และจัดการฐานข้อมูลอ้างอิง
2.  **Qualifier:** กรรมการผู้ใช้งานระบบเพื่อดูเฉลยหรือคุมสอบ
3.  **Trainee:** ผู้รับการทดสอบ (ใช้งานผ่านโหมด Online หรืออ่านจากเล่มคำถาม)

## 4. Technical Reference (แนวทางเทคนิค)
*   **Data Structure:** ใช้โครงสร้างข้อมูลที่สามารถ Map เข้ากับ Component ปัจจุบัน (เช่น `202Lcp.tsx`) ได้โดยตรง
*   **Storage Format:** ใช้ SQLite สำหรับ Structured Data และ File System สำหรับ Media (Images) ตามโมเดล `Hybrid Storage`
