# แผนการปรับปรุงและพัฒนา Backend (Backend Improvement Plan)

เอกสารฉบับนี้รวบรวมแผนงานสำหรับการปรับปรุงประสิทธิภาพ (Optimization), ความปลอดภัย (Security), และโครงสร้างสถาปัตยกรรม (Architecture) ของระบบฐานข้อมูลและ Backend ในโครงการ PQS RTN Hybrid Storage เพื่อให้ระบบมีความเสถียร ซ่อมบำรุงง่าย และพร้อมรองรับการเติบโตในอนาคต

---

## 1. การแบ่งโมดูล (Code Refactoring)
**สถานะ:** ✅ เสร็จสิ้นไปแล้วบางส่วน และจัดทำแผนปฏิบัติการครบถ้วนแล้ว

การแยกไฟล์ "God File" (`content_database.rs`) ที่มีขนาดใหญ่กว่า 8,000 บรรทัด ออกเป็นโมดูลย่อยๆ ตามหลักการ Single Responsibility Principle (SRP)
- **สิ่งที่ทำไปแล้ว:** ความคืบหน้าปัจจุบันและโรดแมปสำหรับการแยกโมดูลทั้งหมดได้ถูกจัดทำเป็นเอกสาร และทำสำเร็จไปแล้วถึง 4 Phase (ลดขนาดไฟล์หลักไปได้เกือบ 40%) 
- **รายละเอียดฉบับสมบูรณ์:** อ้างอิงแผนทั้งหมดได้ที่ `src-tauri/src/content_database/REFACTORING_PROGRESS.md`

---

## 2. การจัดการ Connection (Connection Pooling)
**สถานะ:** 🟡 วางแผน

ปัจจุบันแอปพลิเคชันมีการเปิด Connection ใหม่ (`get_content_connection()`) ทุกครั้งที่มีการเรียกใช้ฟังก์ชันฐานข้อมูล ซึ่งอาจทำให้เกิด Overhead ในการเปิด/ปิดไฟล์ SQLite บ่อยครั้ง และอาจเกิดปัญหา "database is locked" ได้หากมีการดึงข้อมูลพร้อมกันหลายจุด
- **ปัญหาปัจจุบัน:** รูปแบบ `get_content_connection()?` ถูกเรียกซ้ำซากในทุกฟังก์ชัน
- **แนวทางแก้ไข:**
  1. ใช้ **Tauri State Management** (`tauri::State`) ร่วมกับ Connection Pool 라이บรารี เช่น `r2d2_sqlite` หรือ `deadpool_sqlite`
  2. สร้าง Pool ไว้ตั้งแต่ตอนเริ่มแอปพลิเคชัน (App Initialization)
  3. ส่ง Connection Pool ผ่าน Payload/State ไปยังฟังก์ชันต่างๆ เพื่อหมุนเวียนการใช้ Connection เดิม แปรรูปโครงสร้างฟังก์ชันให้รับ `&Connection` หรือ `&Pool` เป็น Parameter

---

## 3. การจัดการความปลอดภัยของข้อมูล (Data Security)
**สถานะ:** 🟡 วางแผน

ฐานข้อมูล SQLite ที่ใช้จัดเก็บข้อมูลหลักและผลคะแนน (PQS Documents, User Answers) ปัจจุบันทำงานในรูปแบบไฟล์เปิด (.db) ซึ่งสามารถถูกคัดลอกหรือแก้ไขข้อมูลได้โดยตรง หากอุปกรณ์ถูกเข้าถึง
- **ปัญหาปัจจุบัน:** ไฟล์ฐานข้อมูลเป็น Plain Text ไม่มีระบบป้องกัน
- **แนวทางแก้ไข:**
  1. **SQLCipher:** เปลี่ยนจาก `rusqlite` ธรรมดาไปใช้ `rusqlite` แบบเปิด feature `sqlcipher` เพื่อทำการเข้ารหัสไฟล์ `.db` ทั้งก้อน (Transparent Data Encryption)
  2. การจัดการกุญแจเข้ารหัส (Key Management): ใช้แนวคิดผูก Encryption Key เข้ากับ Hardware ID หรือใช้ Key ที่สร้างขึ้นแบบสุ่มและเก็บซ่อนอย่างปลอดภัยผ่าน OS Keychain / Tauri Secure Storage
  3. **Data Integrity Checks:** สร้าง hash / checksum signature ในแต่ละ Row หรือสำหรับเอกสาร (ตัวอย่างเช่น ข้อมูลผลสอบหรือผลคะแนน) เพื่อตรวจสอบว่าข้อมูลไม่ได้ถูกดัดแปลง (Tamper Detection) นอกระบบ

---

## 4. การจัดการรุ่นของฐานข้อมูล (Migration System)
**สถานะ:** 🟡 วางแผน

เมื่อแอปพลิเคชันมีการอัปเดตเวอร์ชันและตารางฐานข้อมูลจำเป็นต้องเปลี่ยนโครงสร้าง (Add/Drop Column) หากไม่มีระบบจัดการเวอร์ชันที่ดี จะทำให้แอปเกิดข้อผิดพลาดกับฐานข้อมูลเก่าของผู้ใช้
- **ปัญหาปัจจุบัน:** ฐานข้อมูลใช้การสั่ง PRAGMA โครงสร้างและใช้คำสั่ง DDL โดยตรงในฟังก์ชัน `initialize_content_database()` การทำ Migration ปัจจุบันต้องพึ่งพา script ทิ้งท้ายซึ่งอาจตกหล่นได้ง่าย
- **แนวทางแก้ไข:**
  1. **Framework:** นำเข้า Migration Tool/Framework อย่างเช่น `refinery` หรือ `Merg` เข้ามาใช้
  2. สร้างโฟลเดอร์สำหรับเก็บไฟล์ Migration (เช่น `V1__init_schema.sql`, `V2__add_new_column.sql`)
  3. ระบบจะตรวจสอบ Current Version ในตารางตัวช่วย และทำการทยอย Apply script ที่เหลือโดยอัตโนมัติตอนเปิดโปรแกรม เพื่อรับประกันว่า Database Schema สะท้อนเวอร์ชันของโค้ดล่าสุดเสมอ

---

## 5. การทำความสะอาดโค้ด (Dead Code Cleanup)
**สถานะ:** 🟡 วางแผน

เป็นกระบวนการต่อเนื่องตลอดวงจรการพัฒนา เพื่อลดขนาด Codebase และความซ้ำซ้อน
- **ปัญหาปัจจุบัน:** มีโค้ดที่ไม่ได้ใช้ถูกทิ้งไว้ในระบบ (Legacy functions, ฟังก์ชันที่ถูกทิ้งเพื่อรอการ Refactor หรือถูกแทนที่ด้วยวิธีที่ดีกว่าแล้ว)
- **แนวทางแก้ไข:**
  1. **Rust Warnings:** เปิดใช้งานระบบ Warning และ lints สมบูรณ์ของ Clippy (`cargo clippy`) เพื่อหาตัวแปรหรือฟังก์ชันที่ครอบครองหน่วยความจำแต่ไม่ได้ใช้งาน (`unused_imports`, `dead_code`)
  2. **Refactoring ลบโค้ดซ้ำซ้อน:** ลบฟังก์ชันเก่าของการ Export/Import ก่อนหน้านี้ หากไม่ตรงกับสถาปัตยกรรมใหม่
  3. ควบคุมการทำ Test Case ให้ Cover ทุกโมดูล เพื่อการันตีว่าลบออกไปแล้วระบบยังเป็นปกติ

---

## 6. ประสิทธิภาพการสืบค้น (SQL Optimization)
**สถานะ:** 🟡 วางแผน

เมื่อข้อมูล PQS ในระบบขยายใหญ่ขึ้น การดึงข้อมูลที่มี Relational table ติดกันหลายชั้น อาจเกิดความล่าช้า (N+1 Query Problem)
- **ปัญหาปัจจุบัน:** มีหลายจุดในฟังก์ชันที่เรียก Sub-query หรือวนลูป Query ฐานข้อมูล (เช่น ในขณะดึง `DocumentHierarchy` หรือโครงสร้าง `Sections`) 
- **แนวทางแก้ไข:**
  1. **Indexing:** เพิ่ม Indexes (ดัชนี) ในคอลัมน์ที่ถูกใช้งานใน `WHERE`, `JOIN` หรือ `ORDER BY` บ่อยๆ (เช่น `unit_owner_id`, `parent_id`, `section_id`)
  2. **Eager Loading:** ปรับการดึงข้อมูลที่เคยทำเป็น Query วนลูป (Iterative Querying) เปลี่ยนไปสั่งด้วย `JOIN` หนาๆ ครั้งเดียวดึงมาครบ หรือใช้ CTE (Common Table Expressions) แบบ recursive ไปเลย (ตอนนี้มีการดึง CTE ให้เห็นบ้างแล้ว ควรนำมาประยุกต์ใช้เพิ่มเติม)
  3. ขยายขีดใช้งานแบบ **Prepared Statements** ในฟังก์ชันที่เรียกใช้ถี่ๆ ลดเวลา Parsing Query Text ของ SQLite

---
> **สรุป:** แผนพัฒนาเหล่านี้สามารถจัดเรียงลำดับความสำคัญ (Priority) ได้ตามความสามารถของผลิตภัณฑ์ที่จะจัดส่ง แนะนำให้โฟกัสที่ **ข้อ 1 (Refactoring)** ให้จบเป็นฐานราก จากนั้นขยับไปโฟกัสที่ **ข้อ 2 (Connection Pooling)** เพื่อสร้างความเสถียรในการประมวลผล แล้วค่อยเจาะลึกลงสู่ความมั่นคงทางข้อมูล (Security) ต่อไป
