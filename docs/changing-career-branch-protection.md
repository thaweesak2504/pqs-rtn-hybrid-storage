# Changing Career Branch Protection

เอกสารชุดนี้อธิบายถึงข้อกำหนดและการป้องกันการเปลี่ยน **สาขาอาชีพหลัก/ย่อย (Major/Minor Career Branch)** เพื่อรักษาความถูกต้องของข้อมูลในเอกสาร PQS

## 1. ข้อกำหนดพื้นฐาน (Strict Requirements)
- **Single Branch Policy**: เอกสารหนึ่งฉบับสามารถมีสาขาอาชีพหลัก/ย่อยได้เพียงแบบเดียวเท่านั้น (ตลอดทั้งเล่ม) ไม่สามารถผสมกันได้
- **Constraint Area**: สาขาอาชีพจะถูกบังคับใช้เพื่อเลือกคำถามย่อย (SubQ) ในหัวข้อดังต่อไปนี้:
  - **Section 2xx.2**: ส่วนประกอบ และชิ้นส่วน (Component Description)
  - **Section 2xx.4**: การทำงานตามปกติ (Standard Operating Parameters)
  - **Section 3xx.2 - 3xx.5**: การเตรียมการ, การเริ่มทำงาน, การปฏิบัติ และการหยุดเครื่อง (Operational Procedures)

## 2. กลไกการป้องกัน (Protection Mechanism)
เมื่อผู้ใช้พยายามแก้ไข Metadata เพื่อเปลี่ยนสาขาอาชีพ ระบบจะทำงานดังนี้:

### 2.1 การตรวจสอบความสัมพันธ์ (Usage Validation)
- ระบบจะตรวจสอบว่าในหัวข้อที่กำหนด (2xx.2, 2xx.4, 3xx.2-3xx.5) มีการเลือก SubQ หรือมีการสร้างคำถามที่เชื่อมโยงกับสาขาเดิมไว้แล้วหรือไม่
- หากมีการใช้งานอยู่ ระบบจะแสดงคำเตือน (Warning) อย่างชัดเจน

### 2.2 การแจ้งเตือนและยืนยัน (User Notification)
- **ข้อความเตือน**: *"การเปลี่ยนสาขาอาชีพ จะทำให้คำถามที่เกี่ยวข้องในข้อ 2xx.2, 2xx.4 และ 3xx.2-3xx.5 ถูกลบทั้งหมด และกลับไปอยู่ในสถานะ '(ไม่ต้องอธิบาย)' หรือ '(ไม่ต้องปฏิบัติ)'"*
- ผู้ใช้ต้องยืนยันการตัดสินใจก่อนระบบจะดำเนินการเปลี่ยนแปลง

### 2.3 การล้างข้อมูลและ Reset (Data Clearing & Reset)
หากผู้ใช้ยืนยันการเปลี่ยนสาขาอาชีพ ระบบจะทำการ Reset ข้อที่เกี่ยวข้องกลับสู่สถานะเริ่มต้น (**Default "Exempted"**):
1. **ลบความเชื่อมโยง**: ลบข้อมูลใน `QuestionSubQuestionLinks` ทั้งหมดในหัวข้อเป้าหมาย
2. **ลบเฉลยและคำตอบ**: ลบข้อมูลใน `QuestionAnswerKeys` และ `UserAnswers` ที่เกี่ยวข้อง
3. **Reset Question State**:
   - ตั้งค่า `question_type` เป็น `'exempted'`
   - ตั้งค่า `is_scored = 0` และ `score = 0`
   - ปรับปรุง `display_text` เป็น:
     - **(ไม่ต้องอธิบาย)**: สำหรับหัวข้อใน Section 200
     - **(ไม่ต้องปฏิบัติ)**: สำหรับหัวข้อใน Section 300

## 3. ขั้นตอนการปฏิบัติ (Workflow)
1. **Metadata Modal**: โหลดสถานะการใช้งานสาขาอาชีพเมื่อเปิด Modal
2. **OnChange**: หากพบการเปลี่ยนค่าและมีการใช้งานอยู่ จะเก็บสถานะ "Pending Conflict"
3. **OnSave**: ตรวจสอบการยืนยันของผู้ใช้ -> เรียก Backend Reset -> บันทึกสาขาอาชีพใหม่
4. **Refresh**: สั่ง Refresh หน้าหลักเพื่อให้ข้อมูล SubQ ที่หายไปสะท้อนผลทันที
